# 项目启动与执行方案设计

## 1. 目标

设计一套 Claude Code 驱动的开发执行方案，覆盖从 M0 立项到 M7 试运行的全流程。核心目标：

- 前后端双轨并行，不互相阻塞。
- 以接口契约为唯一耦合点，前端用 mock 独立开发。
- Claude Code 每步先读文档、出清单、等确认再动手。
- 严丝合缝按照 CLAUDE.md、SCHEDULE.md、DESIGN.md 和 docs/ 下的方案文档执行。

## 2. 分工模式

- **你**：负责 UniApp 前端开发 + 整体协调。将接口契约文档传递给后端团队。
- **后端团队**：负责 JEECG Boot 后端开发和后台管理端。
- **Claude Code**：辅助前端开发 + 编写接口契约文档 + 生成测试用例 + 输出任务清单和验收报告。

两个代码仓库独立维护，你手动传递契约文档给后端。

## 3. 整体架构

```text
文档层（已存在）
├── CLAUDE.md · DESIGN.md · SCHEDULE.md
└── docs/education-mobile-mvp/ (00-06)

接口契约层（Claude Code 编写维护）
└── docs/api-contracts/
    ├── 01-登录认证.md
    ├── 02-课程模块.md
    ├── 03-课节播放.md
    ├── 04-学习进度.md
    ├── 05-信息消息.md
    └── 06-页面配置.md

          ↓ 前后端各自按契约开发 ↓

前端轨：UniApp + Vue3                    后端轨：JEECG Boot
① 读契约 → 生成 mock 数据                ① 建表（迁移 SQL）
② 写页面 + 组件 + 业务 API               ② JEECG Online 生成 CRUD
③ 用 mock 自测 UI 和交互                 ③ 实现 App API（/api/app/edu/v1）
④ 后端就绪 → 测试用例验证 → 切换真实 API  ④ 按契约对齐字段和响应格式
```

## 4. 阶段执行协议（SOP）

每个 SCHEDULE.md 阶段按以下步骤执行：

### 步骤 1：阶段启动

- **你**："开始 M<X> 阶段。先读 SCHEDULE.md 第 X.X 节和 <相关文档>，输出本阶段任务清单，等我确认。"
- **Claude Code**：自动读取相关文档 → 提取阶段目标、交付物、验收标准 → 分解为具体任务项 → 区分后端和前端任务 → 输出清单 → 等待确认。

### 步骤 2：接口契约先行

- **你**："根据确认的任务清单，把其中涉及 App API 的接口按契约格式写入 docs/api-contracts/。"
- **Claude Code**：列出所有 App API → 为每个接口写完整契约（路径、参数类型、响应 JSON 示例、字段类型注释、实现状态）→ 写入对应 .md 文件。
- **你**：确认或修正 → 手动传给后端团队。

### 步骤 3：双轨并行开发

前端轨（Claude Code 执行）：
1. 按契约生成 mock 数据到 `mock/` 目录。
2. 读 DESIGN.md 相关 UI 规范。
3. 写页面组件和业务 API 封装。
4. 用 mock 数据自测 UI 交互和流程。

后端轨（后端团队执行）：
1. 建表和执行迁移 SQL。
2. JEECG Online 生成 CRUD。
3. 实现 App API 接口（/api/app/edu/v1）。

### 步骤 4：接口切换验证

- **你**："<模块>后端接口已就绪。根据对应契约文档生成测试用例。我验证通过后切换代码。"
- **Claude Code**：生成测试用例 checklist（字段存在性、类型、参数、边界、响应结构）。
- **你**：执行验证，逐字段对比实际响应与契约。
- **验证结果**：
  - 全部通过 → Claude Code 执行切换代码（mock → 真实 API）。
  - 有失败项 → Claude Code 分析差异来源（后端 bug 或契约需修正）→ 修正后重新验证。

### 步骤 5：阶段验收

- **你**："M<X> 阶段开发完成，对照 SCHEDULE.md 验收标准逐条检查，输出验收报告。"
- **Claude Code**：逐条核对验收标准 → 标注通过/未通过 → 未通过项给出修复建议。

## 5. 接口契约文档格式

每个功能模块一个 `.md` 文件，放在 `docs/api-contracts/` 下。

### 文件结构

```markdown
# <模块名>接口契约

## 基础信息
- **模块**：<模块名>
- **版本**：v1.0.0
- **状态**：🚧 后端未实现 | ✅ 已实现
- **最后更新**：YYYY-MM-DD

## 接口列表

### X.X <接口名称>

<METHOD> /api/app/edu/v1/<path>

**请求参数**
| 参数名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| ... | ... | ... | ... |

**响应示例**
{
  "success": true,
  "code": 200,
  "result": {
    "fieldName": "value"  // 类型 | 可空性 | 说明
  }
}
```

### 格式约束

- 每个响应字段必须标注类型（String/Long/Integer/BigDecimal/Boolean/List/Object）、是否可空、含义。
- 每个接口标注实现状态（🚧 未实现 / ✅ 已实现）。
- 响应结构统一遵循 `{ success, code, message, result }`。
- 分页统一使用 `{ total, pageNo, pageSize, records }`。

## 6. Mock 数据策略

### 目录结构

```text
mock/
├── index.js              // 导出所有 mock，注册到拦截器
├── course/
│   ├── categories.json   // GET /api/app/edu/v1/course/categories
│   ├── list.json         // GET /api/app/edu/v1/course/list
│   └── detail.json       // GET /api/app/edu/v1/course/detail
├── lesson/
│   ├── list.json         // 课节目录
│   └── play-info.json    // 播放信息
├── auth/
│   ├── login.json
│   └── send-code.json
├── user/
│   ├── progress-save.json
│   └── my-courses.json
└── message/
    ├── list.json
    └── unread-count.json
```

### 核心规则

- Mock 文件路径与 API 路径对应，拦截器根据 URL 匹配。
- **Mock 数据严格按契约文档构造**：字段名、类型、嵌套结构与契约完全一致。
- 切换模式：`config.js` 中 `MOCK_MODE = true/false`，全局一处控制。
- 每份 mock 至少 3 条数据：正常数据、空列表、单条数据，覆盖空态和边界。
- Console 打印 `[Mock] GET /api/app/edu/v1/course/list` 便于调试。

### 请求链路

```text
页面组件 → api/ 业务方法 → utils/request.js → 拦截器判断 MOCK_MODE
  → true:  匹配 mock/ 目录 JSON → 模拟 200ms 延迟返回
  → false: 正常 HTTP 请求 → Token 注入 → 统一错误处理
```

## 7. 测试用例策略

### 切换流程

```text
Claude Code 生成测试用例 → 你执行字段验证 → 全部通过 → Claude Code 切换代码
                                    → 有失败 → Claude Code 分析差异 → 修正后重验
```

### 测试用例覆盖五层

1. **字段存在性检查**：逐字段对比实际响应与契约（字段路径、契约类型、实际返回、通过/失败）。
2. **参数验证**：分页参数、筛选参数、排序参数是否生效。
3. **过滤条件验证**：分类筛选、关键词搜索、状态过滤。
4. **边界与异常**：空数据、非法参数、超大值、已下线内容不出现。
5. **响应结构统一性**：success/code/result 结构、分页字段、错误码和错误信息格式。

### 必测 P0 回归项

- 已授权课程可播放。
- 未授权课程不可播放。
- 已下线课程不在列表中。
- 已下线信息不在列表中。
- 学习进度保存后可恢复。
- 消息未读数正确变化。

## 8. Claude Code 指令模式

### 指令三要素

每次指令必须包含：
1. **指定文档**：明确先读哪个文件的哪一节。
2. **明确产出**：要清单/契约/代码/测试用例/验收报告？
3. **设定确认点**："等我确认" / "我验证后再" 明确但不越权。

### 五大场景指令模板

**场景一：阶段启动 → 输出任务清单**

> "开始 M<X> 阶段。先读 SCHEDULE.md 第 X.X 节和 <相关文档>，输出本阶段任务清单，区分后端任务和前端任务，等我确认。"

**场景二：接口契约 → 编写并输出文档**

> "根据刚才确认的任务清单，把其中涉及 App API 的接口按契约文档格式写入 docs/api-contracts/。每个接口标注字段类型、可空性、响应示例和实现状态。"

**场景三：前端开发 → Mock 并行**

> "根据 docs/api-contracts/<模块>.md 的接口契约，为 <页面> 生成 mock 数据，然后实现 <页面>。参考 DESIGN.md 的 <相关章节> UI 规范，先读文档再动手。"

**场景四：接口切换 → 测试用例 + 切换代码**

> "<模块> 后端接口已就绪。根据 docs/api-contracts/<模块>.md，生成 <接口列表> 的测试用例。我验证通过后，你切换代码从 mock 到真实 API。"

**场景五：阶段验收 → 对照标准逐条检查**

> "M<X> 阶段开发完成，现在做阶段验收。对照 SCHEDULE.md 第 X.X 节的验收标准，逐条检查并输出验收报告，标注通过和未通过项。"

## 10. 关键约束

- P2 功能（AI、题库、商城、社区、直播、支付、积分）不进入首期。若被提及，先确认是否调整范围。
- 不在页面里直接写复杂业务请求，业务 API 放在 `api/` 目录。
- 路由跳转统一封装，不在组件内拼散乱路径。
- 页面配置只允许白名单组件（banner/noticeBar/imageNav/courseList/infoList/topicCard/richText/spacer）。
- 严禁执行后台下发的任意 JavaScript、远程脚本、HTML 注入。富文本必须安全过滤。
- 不做无关重构，不引入与 JEECG/UniApp 体系冲突的大型框架。
