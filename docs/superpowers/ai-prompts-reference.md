## 2. 接口契约 → 编写文档

```text
根据刚才确认的任务清单，把其中涉及 App API 的接口按契约文档格式写入 docs/api-contracts/。每个接口标注字段类型、可空性、响应示例和实现状态。
```

---

## 3. 前端开发 → Mock 并行

```text
根据 docs/api-contracts/<模块>.md 的接口契约，为 <页面名> 生成 mock 数据，然后实现 <页面名>。参考 DESIGN.md 的 <相关章节> UI 规范，先读文档再动手。
```

**示例：**
```text
根据 docs/api-contracts/02-课程模块.md 的接口契约，为课程列表页生成 mock 数据，然后实现课程列表页。参考 DESIGN.md 第 9 节和第 12.2 节课程卡规范，先读文档再动手。
```

---

## 4. 接口切换 → 测试用例 + 切换代码

```text
<模块名> 后端接口已就绪。根据 docs/api-contracts/<模块>.md，生成 <接口1>、<接口2> 的测试用例。我验证通过后，你切换代码从 mock 到真实 API。
```

**示例：**
```text
课程模块后端接口已就绪。根据 docs/api-contracts/02-课程模块.md，生成课程分类列表、课程列表、课程详情三个接口的测试用例。我验证通过后，你切换代码从 mock 到真实 API。
```

## 5. 阶段验收 → 逐条检查

```text
M<X> 阶段开发完成，现在做阶段验收。对照 SCHEDULE.md 第 <X.X> 节的验收标准，逐条检查并输出验收报告，标注通过和未通过项。
```

## 6. 临时/补充指令

### 只读文档不写代码
```text
先读 <文件路径>，总结关键信息，等我确认后再动手。
```

### 只改一个文件
```text
修改 <文件路径> 的 <具体内容>，不要动其他文件，改完等我 review。
```

### 修复验收失败项
```text
验收报告中有 <N> 项未通过。逐项分析原因并修复，每修一项等我确认。
```

### 补充某个缺失功能
```text
检查 <功能名> 是否已实现。如果没有，输出实现计划，等我确认。
```
