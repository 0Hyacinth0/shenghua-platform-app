# CLAUDE.md

本文件是本项目的协作与开发约束，供后续开发者和代码代理读取。请优先遵守本文件，再参考 `DESIGN.md`、`SCHEDULE.md` 和 `docs/education-mobile-mvp/` 下的详细方案。

## 1. 项目目标

构建一款教育类移动应用 MVP。首期只做：

- 手机号登录和隐私协议确认。
- 首页、课程列表、课程详情、视频课节播放。
- 我的课程、学习进度、断点续播。
- 后台课程、课节、视频、信息、消息管理。
- 后台通过 JEECG 页面编辑能力配置首页/专题等运营 UI。
- UniApp 端拉取已发布 schema 并用白名单组件渲染。

不要在首期实现 AI、题库、商城、社区、直播、积分、优惠券、复杂推荐、讲师端。

## 2. 已确认技术选型

| 层级 | 选型 |
| --- | --- |
| 后端 | JEECG Boot / Spring Boot 3 分支 |
| 后台前端 | JEECG Vue3 管理端 |
| ORM | MyBatis-Plus |
| 数据库 | MySQL 8 |
| 缓存 | Redis |
| 移动端 | UniApp + Vue3 |
| 视频 | 对象存储或云点播 + CDN |
| API 文档 | JEECG/SpringDoc/Knife4j，随项目实际脚手架确定 |

## 4. 范围控制

实现功能前先判断优先级：

- P0：必须实现，影响 MVP 是否能上线。
- P1：首期增强，可以随第一个小版本上线。
- P2：二期候选，不要主动实现。

如果用户提出类似“AI 学习、题库、商城、社区、直播、支付、积分”的需求，先确认是否要调整首期范围，不要直接扩展。

## 5. 后端开发约束

- 新增教育业务建议放在 `jeecg-module-edu`。
- 业务表使用 `edu_` 前缀。
- 保留 JEECG 标准字段：`create_by`、`create_time`、`update_by`、`update_time`、`del_flag`。
- 常规 CRUD 优先使用 JEECG Online 表单/代码生成。
- 发布、下线、授权、回滚等状态变更必须写业务校验和操作日志。
- 移动端 API 不直接返回后台实体全字段，应使用 App DTO/VO。
- 课程、课节、信息接口必须过滤状态，不能返回已下线内容。
- 课节播放接口必须校验课程是否公开、试看或用户已授权。
- 页面配置必须持久化到数据库，Redis 只能做缓存。

## 6. UniApp 前端开发约束

- 使用 UniApp + Vue3。
- 页面注册、全局窗口样式、底部 Tab 通过 `pages.json` 管理。
- 底部 Tab 首期固定为：首页、课程、消息、我的。
- 所有接口通过统一 request 封装调用，统一处理 Token、401、错误提示和 loading。
- 本地缓存用于隐私协议状态、Token、首页 schema、最近学习位置。
- 视频播放页要单独做真机验证。UniApp 的 video 在非 H5 平台可能是原生组件，普通前端层级不一定能覆盖。
- 路由跳转统一封装，不在组件内拼散乱路径。
- 不在页面里直接写复杂业务请求，业务 API 放在 `api/` 目录。

## 7. 配置化 UI 约束

后台页面编辑器只允许影响内容型 UI：

- 首页运营区。
- 活动页。
- 专题页。
- 信息聚合页。

禁止用页面编辑器控制：

- 登录流程。
- 播放器核心逻辑。
- 课程授权判断。
- 学习进度上报。
- 消息已读逻辑。

App 只渲染白名单组件：

- `banner`
- `noticeBar`
- `imageNav`
- `courseList`
- `infoList`
- `topicCard`
- `richText`
- `spacer`

严禁执行后台下发的任意 JavaScript、远程脚本、任意 HTML 注入。富文本必须做安全过滤。

## 8. 页面配置发布要求

页面配置必须支持：

- 草稿保存。
- 移动端预览。
- 版本发布。
- 历史版本。
- 一键回滚。
- 测试和生产环境隔离。
- Redis 缓存刷新。
- App 本地缓存降级。
- 旧 App 版本兼容。

保存和发布时都要做 schema 校验。未知组件在 App 端应跳过并记录日志，不能导致白屏。

## 9. 接口规范

移动端接口前缀：

```text
/api/app/edu/v1
```

后台业务接口前缀：

```text
/api/admin/edu/v1
```

响应结构沿用 JEECG `Result` 风格：

```json
{
  "success": true,
  "code": 200,
  "message": "操作成功",
  "result": {},
  "timestamp": 1770000000000
}
```

分页参数统一使用：

```text
pageNo
pageSize
```

## 10. 开发流程

- 先更新接口文档，再开发接口和页面。
- 前后端并行开发时使用 mock 数据，但字段名必须与接口文档一致。
- 每个 P0 功能必须有验收标准。
- 修改数据库结构时同步提交迁移 SQL。
- 修改页面 schema 协议时同步更新 App 渲染器、后台校验和文档。
- 不做无关重构，不引入与 JEECG/UniApp 体系冲突的大型框架。

## 11. 测试重点

P0 回归必须覆盖：

- 首次启动隐私协议。
- 手机号登录。
- 首页 schema 拉取失败后的默认首页。
- 课程列表、课程详情、课节播放。
- 未授权课程不能播放。
- 学习进度保存和断点续播。
- 后台发布信息后 App 展示。
- 消息未读和已读。
- 页面配置保存、预览、发布、回滚。
- 旧版本 App 遇到未知组件不崩溃。

## 12. 排期来源

功能优先级、阶段排期、里程碑和延期风险统一以 `SCHEDULE.md` 为准。不要在 `DESIGN.md` 或本文件中维护另一份排期，避免团队执行时出现多个版本。

## 13. 代码代理工作方式

在后续实现中：

- 做 UI 页面或组件前，先读 `DESIGN.md`。
- 做功能拆分、排期判断或版本范围判断前，先读 `SCHEDULE.md`。
- 做接口、数据表或后台实现前，先读相关 `docs/education-mobile-mvp/*.md`。
- 先检查现有代码结构，再新增文件。
- 遵循 JEECG 和 UniApp 既有目录与命名。
- 修改前说明将改哪些文件。
- 修改后运行可用的测试、构建或静态检查。
- 不要删除用户已有改动。
- 不要主动提交、推送或创建 PR，除非用户明确要求。

## 14. Claude Code 开发执行流程

本项目的开发由 Claude Code 按以下协议驱动。完整设计见 `docs/superpowers/specs/2026-05-27-project-startup-and-execution-design.md`。

### 14.1 分工

- **你（用户）**：负责 UniApp 前端开发 + 整体协调。将接口契约文档手动传递给后端团队。
- **后端团队**：负责 JEECG Boot 后端开发和后台管理端。
- **Claude Code**：辅助前端开发 + 编写接口契约文档 + 生成测试用例 + 输出任务清单和验收报告。

两个代码仓库独立维护。

### 14.2 阶段开发五步协议

每个 SCHEDULE.md 阶段（M0-M7）按以下步骤执行，不可跳过：

1. **阶段启动出清单**：Claude Code 先读 SCHEDULE.md 对应章节和相关文档 → 输出任务清单（区分后端和前端）→ 等你确认。
2. **接口契约先行**：Claude Code 按统一格式编写 `docs/api-contracts/<模块>.md`（字段类型、可空性、响应示例、实现状态）→ 你确认 → 传给后端。
3. **双轨并行开发**：前端按契约生成 mock 数据写页面，后端建表实现 API，互不阻塞。
4. **测试用例驱动切换**：后端接口就绪后 → Claude Code 生成测试用例 → 你执行验证 → 全部通过后 Claude Code 切换 mock 到真实 API。
5. **阶段验收**：Claude Code 对照 SCHEDULE.md 验收标准逐条检查，输出验收报告。

### 14.3 Claude Code 指令三要素

每次给 Claude Code 下指令必须包含：

1. **指定文档**：明确先读哪个文件的哪一节。
2. **明确产出**：要清单 / 契约 / 代码 / 测试用例 / 验收报告？
3. **设定确认点**："等我确认" / "我验证后再" 但不要越权。

### 14.4 五大场景指令模板

**场景一：阶段启动 → 输出任务清单**

> "开始 M<X> 阶段。先读 SCHEDULE.md 第 X.X 节和 <相关文档>，输出本阶段任务清单，区分后端任务和前端任务，等我确认。"

**场景二：接口契约 → 编写并输出文档**

> "根据刚才确认的任务清单，把其中涉及 App API 的接口按契约文档格式写入 docs/api-contracts/。每个接口标注字段类型、可空性、响应示例和实现状态。"

**场景三：前端开发 → Mock 并行**

> "根据 docs/api-contracts/<模块>.md 的接口契约，为 <页面> 生成 mock 数据，然后实现 <页面>。参考 DESIGN.md 的 <相关章节> UI 规范，先读文档再动手。"

**场景四：接口切换 → 测试用例 + 切换代码**

> "<模块> 后端接口已就绪。根据 docs/api-contracts/<模块>.md，生成 <接口列表> 的测试用例。我验证通过后，你切换代码从 mock 到真实 API。"

**场景五：阶段验收 → 对照标准逐条检查**

> "M<X> 阶段开发完成，现在做阶段验收。对照 SCHEDULE.md 第 X.X 节的验收标准，逐条检查并输出验收报告，标注通过和未通过项。"

### 14.6 Mock 与测试约束

- Mock 数据放在 `mock/` 目录，按 API 路径组织，字段名、类型、嵌套结构与契约严格一致。
- 每份 mock 至少包含正常数据、空列表、单条数据三种情况。
- 全局 Mock 开关在 `config.js` 中 `MOCK_MODE` 字段，一处切换。
- 接口测试用例覆盖五层：字段存在性、参数验证、过滤条件、边界异常、响应结构统一性。
- P0 回归必测：授权播放、未授权拦截、已下线过滤、进度恢复、消息未读数。
